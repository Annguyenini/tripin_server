How service handle users credential:
Signup: (/auth/signup)
clients required to send email, displayname, username, password
Then server check for duplicate email or username,
Server then send a comfirmation email with 6 digits code.
Client required to verify the code. (/auth/verify-code)

Server than process to store user data into postgresQL, with hash passwords.
Server redirect clients to login page.

Login: (/auth/login)
User send username and password, the server then verify the credentials.
After successfully verify, server start processing 2 tokens.
One token (access token) with exp time = 1hr
One token (refresh token) with exp time 1 month
refresh token will be store in postgresQL

Login-Via_token:(/auth/login_via_token)
1st senario: user still have valid access key, server will return code 200 and user been able to access their data, user will be ablt to skip the credentials fill out required
2nd senario: user have unvalid or expired access key, but valid refresh key, server then generate new acces token and send back to user, user still be able to skip the credentials fill out requirement.
3st senario: user have expired refresh token, server then will revoken the expired token in postgresQL then return status code 400 and requuire users to fill out credentials.
4st senario: user doesn't have valid access token and refresh token, the server will then return status code 400 and require users to fill out credentials.

request-new-access-token: (/auth/request_access_token)
For Login-Via-Token senario 2, after sever return code 400, clients will send another request for new access-token
